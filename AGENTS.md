# AGENTS.md

This repository is an Astro + Starlight documentation site. Use this file as the default operating manual for any AI agent creating, editing, or customizing the site.

This guidance is based on the repo state and the official Astro/Starlight docs reviewed on 2026-03-17.

## Project Snapshot

- Package manager: `yarn`
- Framework: `astro@^6.0.1`
- Docs theme/framework: `@astrojs/starlight@^0.38.1`
- Content collection config lives in `src/content.config.ts`
- Main site config lives in `astro.config.mjs`
- Docs content lives in `src/content/docs/`

Current repo specifics:

- `src/content.config.ts` defines a single `docs` collection using `docsLoader()` and `docsSchema()`. Keep this unless the task explicitly requires a different content model.
- `astro.config.mjs` is configured around a single autogenerated `Research` sidebar group using `autogenerate: { directory: 'research' }`
- `src/content/docs/index.mdx` is a splash homepage
- The primary published content currently lives in `src/content/docs/research/`. Do not rewrite, delete, or relocate that material unless the user asks.
- Some research notes may still be working drafts rather than fully normalized Starlight pages. Do not mass-convert them unless the user asks for that cleanup.

## Default Workflow

1. Read `astro.config.mjs`, `src/content.config.ts`, and the files you plan to edit before making changes.
2. Prefer the smallest change that fits Starlight's built-in model.
3. Prefer configuration and content changes before adding custom components, plugins, or new tooling.
4. After meaningful site changes, run `yarn build` if dependencies are available and verification will not be polluted by unrelated local work.

## Where Changes Belong

- Docs pages: `src/content/docs/**/*.md` or `src/content/docs/**/*.mdx`
- Collection config: `src/content.config.ts`
- Site-wide Starlight config: `astro.config.mjs`
- Local optimized images used by docs or config: `src/assets/`
- Static files like favicons: `public/`
- Custom CSS: `src/styles/*.css`, then register it in `starlight({ customCss: [...] })`
- Custom or override components: `src/components/`
- Fully custom Astro routes: `src/pages/`

Do not put normal docs pages in `src/pages/` unless you intentionally need a fully custom route or a data-driven page outside the docs collection.

## Content Authoring Rules

- Use `.md` by default.
- Use `.mdx` only when you need imports, components, or JSX-like markup.
- Every new Starlight content page should include frontmatter at minimum:

```yaml
---
title: Page Title
description: One-sentence summary of the page.
---
```

- Starlight uses the page `title` as the top-level heading. Start the body with normal paragraph text, then use `##` and `###` headings inside the body. Do not add a second `#` heading in page content.
- Prefer site-path internal links like `/research/threat-model-and-value/` instead of hardcoded production domains. If the project later uses Astro `base`, keep internal links consistent with that deployment setup.
- Include alt text on every image.
- Prefer Starlight asides over ad hoc callout formatting:
  - `:::note`
  - `:::tip`
  - `:::caution`
  - `:::danger`
- Use fenced code blocks with language identifiers.
- Use Starlight's documentation patterns intentionally:
  - task-oriented material belongs in `guides/`
  - factual/API/spec material belongs in `reference/`
  - research or analysis material should stay in a clearly named section such as `research/`
- When editing legacy working notes, preserve the user's existing structure unless the task explicitly includes normalizing them into polished Starlight pages.

## Markdown vs. MDX

Choose `.md` when plain content is enough.

Choose `.mdx` when you need:

- built-in Starlight components like cards, tabs, or steps
- custom Astro components
- richer landing pages

Useful built-in MDX components come from `@astrojs/starlight/components`.

Prefer these components when they fit:

- `<Card>` and `<CardGrid>` for grouped highlights or landing-page summaries
- `<Steps>` for multi-step task flows
- `<Tabs>` and `<TabItem>` for equivalent option sets like OS/package-manager variants

If multiple tab groups on a page should stay aligned, use the same `syncKey`.

If you create a custom component that should not inherit doc-prose styling, wrap it in an element with the `not-content` class.

## Navigation And Sidebar Rules

The current repo uses a single autogenerated `Research` sidebar section.

- New files in `src/content/docs/research/` appear automatically because the sidebar uses `autogenerate: { directory: 'research' }`.
- If the user later adds other top-level content areas such as `guides/` or `reference/`, decide explicitly whether they should be their own autogenerated sections or manual sidebar groups.
- Do not recreate starter `guides/` or `reference/` folders unless the information architecture actually needs those sections.

For autogenerated sidebar groups, page frontmatter `sidebar` metadata can control:

- `label`
- `order`
- `hidden`
- `badge`
- `attrs`

Use manual sidebar entries when you want tight control over sequence or grouping. Use autogenerated groups when directory structure should drive navigation.

Do not add manual sidebar `slug` entries pointing at draft pages. Draft pages are excluded from production builds.

## Frontmatter To Use Intentionally

These are the Starlight frontmatter fields most likely to matter on this project:

- `title`
- `description`
- `slug`
- `editUrl`
- `head`
- `tableOfContents`
- `template`
- `hero`
- `banner`
- `lastUpdated`
- `prev`
- `next`
- `pagefind`
- `draft`
- `sidebar`

Use them with intent:

- `template: splash` is appropriate for landing pages, not normal docs pages.
- `hero` belongs mainly on splash pages like the homepage.
- `tableOfContents: false` is reasonable for short pages or landing pages.
- `pagefind: false` is useful for private, duplicate, or low-value search pages.
- `draft: true` keeps a page visible in development but excludes it from production builds.
- `prev` and `next` are for overriding pagination when the default sequence is not right.

## Homepage Rules

`src/content/docs/index.mdx` is currently the homepage and uses `template: splash`.

When changing the homepage:

- keep it in `src/content/docs/index.mdx` unless there is a clear reason to move to a custom route
- use splash layout only if the page should act like a marketing or landing page
- if the homepage should behave like normal documentation, remove `template: splash`

## Styling And Theming

Prefer this order of operations:

1. Change Starlight config in `astro.config.mjs`
2. Add or adjust custom CSS through `customCss`
3. Override components only if config and CSS are not enough
4. Add plugins only when a site-wide extension is truly needed

Styling guidance:

- Add custom CSS under `src/styles/` and register it with `customCss`
- Prefer CSS custom properties and small targeted overrides before writing large selector-heavy CSS
- Starlight uses cascade layers internally; unlayered custom CSS will override Starlight defaults
- If you use layers, place your override layer after `starlight`
- Use `expressiveCode` config for code-block theming and behavior instead of hacking code-block DOM with brittle CSS

Tailwind:

- Tailwind is not installed in this repo today
- Do not add Tailwind casually for one-off styling tweaks
- If the user explicitly wants Tailwind, follow the Starlight Tailwind setup and make the Tailwind CSS entry the first item in `customCss`

## Component Overrides

Only override Starlight components when configuration and CSS cannot achieve the desired result.

When overriding:

- place custom components in `src/components/`
- wire them through `starlight({ components: { ... } })` in `astro.config.mjs`
- prefer wrapping a built-in Starlight component over fully replacing it
- preserve `<slot />` and any named slots when wrapping built-in components
- remember that overrides apply globally

If an override should affect only some pages, branch on `Astro.locals.starlightRoute` inside the component instead of creating multiple override systems.

## Custom Pages

Use `src/pages/` only for advanced cases:

- custom layouts not driven by docs content
- dynamic or alternative data sources
- specialized routes like custom 404s, API-like pages, or generated indexes

If you need a custom page that still looks like Starlight, wrap it with `@astrojs/starlight/components/StarlightPage.astro`.

Keep in mind:

- content in `src/content/docs/` participates naturally in Starlight docs behavior
- pages in `src/pages/` are regular Astro routes and are not part of autogenerated docs collections

## Content Schema Rules

Keep the current docs collection shape unless there is a strong reason to change it:

```ts
docs: defineCollection({ loader: docsLoader(), schema: docsSchema() })
```

If the user needs extra frontmatter fields:

- extend `docsSchema({ extend: ... })`
- do not replace Starlight's schema from scratch unless the task truly requires it

Avoid adding a second docs-like collection without a clear architectural reason.

## Deployment-Aware Config

Before production deployment:

- set Astro `site` to the canonical production URL
- set Astro `base` if the site is served from a subpath
- make sure custom asset URLs respect `import.meta.env.BASE_URL` when `base` is used

Do not hardcode the final domain into normal internal doc links.

## Repo-Specific Cleanup Targets

When the user asks for further customization, likely next targets are:

- set Astro `site` in `astro.config.mjs`
- replace the remaining generic starter asset choices if they are still present
- add missing frontmatter like `description` or `sidebar.order` to research pages where helpful
- decide whether to add non-research sections beyond the current `research/` library
